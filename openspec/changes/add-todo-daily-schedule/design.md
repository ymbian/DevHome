## Context

当前平台前端由 `platform/src/main/resources/static/` 下的静态页面组成，侧边栏导航由各页面 HTML 和 `shell.js` 共同实现。现有导航只覆盖“开发”“技改”，缺少对每日执行计划的全局视图。  
本次目标是快速提供可感知的待办事项模块，以 mock 数据验证信息密度、状态表达和可读性，不引入后端接口与数据库改造。

## Goals / Non-Goals

**Goals:**
- 在全站侧边栏新增“待办事项”入口，保持导航体验一致。
- 新增待办事项页面，覆盖 00:00-23:00 小时段展示。
- 用清晰视觉状态表达任务完成情况：
  - 已完成：绿色对勾
  - 已过截止且未完成：红色
  - 其余：默认状态
- 使用前端 mock 数据直接渲染，页面打开即可看到完整示例。

**Non-Goals:**
- 不接入数据库，不新增持久化模型。
- 不新增后端 API，不实现多用户同步。
- 不做复杂排班算法、拖拽编排或提醒推送。

## Decisions

### 1) 采用纯前端 mock 渲染，不新增后端接口
- **Decision**: 在 `todo.html` 中维护 mock 数据并渲染。
- **Rationale**: 需求核心是“可视化一目了然”，先验证 UI/交互价值；后端化会引入额外复杂度。
- **Alternative considered**: 新增 `/api/todos` 接口。被拒绝，因为当前阶段不需要数据持久化。

### 2) 统一按 24 小时固定时段展示
- **Decision**: 页面固定展示 24 个小时段（00-23），每个时段可有 0..N 个任务。
- **Rationale**: 固定时间轴能快速建立空间记忆，用户扫一眼就知道全天安排密度。
- **Alternative considered**: 仅展示有任务的时段。被拒绝，因为无法直观看出“空闲时段”和“全天覆盖范围”。

### 3) 状态计算由“任务完成标记 + 当前时间”决定
- **Decision**:
  - `done = true` -> 显示绿色对勾；
  - `done = false && now > task.end` -> 标红为超时；
  - 其他 -> 正常待办样式。
- **Rationale**: 规则简单、可解释，能消除额外状态分支和手工维护负担。
- **Alternative considered**: 显式维护 `OVERDUE` 状态字段。被拒绝，因为 mock 阶段会造成冗余和状态不一致风险。

### 4) 导航改造采用“局部静态更新 + 高亮规则扩展”
- **Decision**: 在现有静态页面导航加入 `/todo.html` 链接，并在 `shell.js` 增加 `todo` 高亮匹配。
- **Rationale**: 与现有站点结构一致，改动范围可控。
- **Alternative considered**: 抽取统一导航模板。被拒绝，因为当前页面数量较少，先做最小可交付改动。

## Risks / Trade-offs

- [风险] 多页面导航可能漏改，导致入口不一致  
  → Mitigation: 列出所有静态页面逐一补齐导航并手工回归。

- [风险] 颜色语义不一致影响可读性  
  → Mitigation: 固定状态色规范（完成=绿色，超时=红色，其他中性）。

- [权衡] mock 数据无真实业务价值沉淀  
  → Mitigation: 为后续 API 对接预留数据结构字段（id、title、startHour、endHour、done）。

- [权衡] 固定 24 小时视图在移动端信息密度较高  
  → Mitigation: 使用可折叠或简洁卡片样式（后续迭代），本次先保证桌面端一目了然。
