## Context

DevHome 平台已有「开发模块」：需求输入 → 设计生成 → openspec 集成 → 编码就绪（见已归档的 requirement-to-code-platform）。技改需求（漏洞修复、组件升级、数据库规范/性能优化等）目前无统一入口，本设计在现有 Spring Boot 平台基础上增加技改模块，并复用开发模块流水线。

## Goals / Non-Goals

**Goals:**

- 技改任务可持久化、按类型筛选与列表展示。
- 用户可选择一条或多条技改任务「处理」，系统将其转为需求描述并调用开发模块（创建 pipeline、生成设计、运行 openspec），实现技改→openspec→开发全流程。
- 开发模块暴露可供技改调用的 API（创建 pipeline、触发设计/openspec），不改变现有 Web 使用方式。

**Non-Goals:**

- 不实现漏洞/依赖的自动扫描与导入（首版支持手工录入或配置；对接扫描为后续迭代）。
- 不改变 openspec 或 Cursor 的既有用法。

## Decisions

1. **技改任务存储**  
   - **选择**：新增实体 `TechImprovementTask`（类型、标题、描述、优先级、状态、来源等），与 `PipelineInstance` 同库（JPA/H2）。  
   - **理由**：与现有平台一致，查询与关联简单。  
   - **备选**：独立微服务或外部系统对接，首版不采纳。

2. **技改类型**  
   - **选择**：枚举或配置固定类型：漏洞修复、组件版本升级、数据库规范整改、数据库性能优化；支持「其他」以扩展。  
   - **理由**：满足需求中的几类场景，同时保留扩展。

3. **派发到开发模块**  
   - **选择**：技改派发时调用现有后端 API：POST /api/requirements（创建 pipeline）→ 对返回的 instanceId 依次调用设计生成、openspec 运行；可选一次派发创建一条 pipeline（多条技改合并为一条需求描述）或一条技改一条 pipeline。  
   - **理由**：复用现有接口，无需复制业务逻辑。  
   - **约定**：首版采用「每条技改派发对应一个 pipeline 实例」，需求描述 = 技改标题 + 描述 + 类型说明。

4. **前端结构**  
   - **选择**：新增技改入口（如 /tech 或导航「技改」），页面含任务列表、类型筛选、多选与「处理」按钮；处理成功后跳转至开发模块的设计页或实例详情。  
   - **理由**：与现有首页/设计页并列，结构清晰。

5. **开发模块可调用性**  
   - **选择**：现有 POST /api/requirements、POST /api/instances/{id}/design/generate、POST /api/instances/{id}/openspec/run 已满足「创建实例 + 设计 + openspec」的链式调用；技改派发服务层依次调用上述 API 即可，无需新增开发模块接口（仅确保现有 API 可被内部/后端调用）。  
   - **理由**：最小改动，开发模块保持 REST 契约即可。

## Risks / Trade-offs

- **[技改数据来源]** → 首版仅支持手工录入或批量导入配置；后续再对接扫描/工单系统。
- **[派发失败]** → 任一步失败时记录状态并提示用户，支持对同一技改任务重试派发。
- **[幂等与并发]** → 同一技改多次「处理」可产生多个 pipeline 实例（可接受）；若需「已处理」防重复，可在技改任务上增加「已关联 instanceId」状态。

## Migration Plan

- 在 platform 模块内新增实体、Repository、Controller、Service 及静态页面；数据库通过 JPA ddl-auto 新增表。
- 部署：无数据迁移；重启后新表生效，技改任务为空直至录入或导入。
- 回滚：移除技改相关代码与表，不影响开发模块现有功能。

## Open Questions

- 技改任务「来源」字段是否需与外部系统 ID 对接（如 Jira key），以便后续同步状态。
- 多选派发时是否支持「合并为一条需求描述、一个 pipeline」的选项。
