## Why

技改任务列表当前来自平台内置存储（如 H2），且字段为通用 title/description/priority。实际运营中任务数据常已在 MySQL 中按类型分表存在，每条记录包含应用名、整改问题描述、整改截止时间等。需要让技改页在切换「漏洞修复」「版本升级」等 Tab 时，从 MySQL 对应表中查询并展示这些记录，与现有数据源对齐。

## What Changes

- **数据源**：技改任务列表改为从 **MySQL** 读取；每种技改类型对应数据库中**一张表**（如漏洞修复表、组件升级表等），表结构包含至少：**应用名**、**整改问题描述**、**整改截止时间**（及主键、可选创建时间等）。
- **API**：保留按 type 筛选的列表接口语义（如 `GET /api/tech-improvement/tasks?type=xxx`），后端根据 type 查询对应 MySQL 表，返回记录列表；响应字段包含应用名、整改问题描述、整改截止时间等，供前端展示。
- **前端**：技改页切换 Tab（漏洞修复、组件版本升级等）时，请求上述接口并展示返回的任务列表；列表项展示**应用名**、**整改问题描述**、**整改截止时间**（及可选类型、操作按钮）。
- **兼容与范围**：可与现有「新增任务」「处理选中」并存：新增可仍写平台库或写入 MySQL 对应表；派发逻辑可沿用或扩展为对 MySQL 记录创建 pipeline。本变更聚焦「列表从 MySQL 按类型分表查询并展示」。

## Capabilities

### New Capabilities

- `tech-improvement-mysql-storage`: 技改任务按类型分表存储于 MySQL；每种类型一张表，表结构包含应用名、整改问题描述、整改截止时间；提供按类型查询列表的接口契约。

### Modified Capabilities

- `tech-improvement-catalog`: 任务列表数据来源为 MySQL 按类型分表；列表展示字段包含应用名、整改问题描述、整改截止时间；切换类型 Tab 时请求接口并展示对应表数据。

## Impact

- **后端**：新增 MySQL 数据源配置；按类型定义实体/表或动态表名映射，JPA 或 JDBC 查询对应表；现有技改列表 API 改为从 MySQL 按 type 查表并返回统一 DTO（含应用名、整改问题描述、整改截止时间）。
- **前端**：tech.html 任务列表区域渲染字段调整为应用名、整改问题描述、整改截止时间；Tab 切换时仍调用现有列表 API（type 参数），仅展示结构变化。
- **依赖**：MySQL 驱动、连接配置；需约定或配置各类型与表名映射（如 漏洞修复 → vulnerability_fix_tasks）。
- **兼容**：若保留现有「新增任务」表单，需约定新增记录写入 MySQL 对应表或保留写平台库二选一；派发可后续迭代对接 MySQL 记录 ID。
