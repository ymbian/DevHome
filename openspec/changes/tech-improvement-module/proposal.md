## Why

当前平台仅有「需求描述 → 设计 → openspec → 开发」的开发模块，缺少针对技术债务与合规的技改入口。漏洞修复、组件版本升级、数据库规范整改、数据库性能优化等技改需求往往散落在工单或表格中，无法统一查看、选择处理，也无法复用现有的「需求→openspec→开发」流水线。本变更为平台增加技改模块，使技改需求可被查看与选择处理，并自动调用开发模块完成从技改需求到 openspec 再到开发的全过程。

## What Changes

- **开发模块定位**：将现有「需求输入 → 设计生成 → openspec 集成 → Cursor 编码」能力明确归为「开发模块」，作为可被其他模块调用的流水线。
- **技改模块新增**：增加技改模块，支持技改任务（漏洞修复、组件版本升级、数据库规范整改、数据库性能优化等）的列表展示、筛选与查看。
- **选择处理与派发**：用户可在技改模块中勾选要处理的技改需求，触发「生成需求描述 → 调用开发模块」的流程，实现技改需求到 openspec 与开发的全过程。
- **技改任务数据**：技改任务需有来源（如手工录入、配置列表或后续对接扫描结果），支持类型、标题、描述、优先级等基础字段，便于筛选与派发。

## Capabilities

### New Capabilities

- `tech-improvement-catalog`: 技改任务目录：支持按类型（漏洞修复、组件版本升级、数据库规范整改、数据库性能优化等）查看、筛选与展示技改任务列表；任务具备基础属性（类型、标题、描述、优先级、状态等）。
- `tech-improvement-dispatch`: 技改任务派发：用户可选择一条或多条技改需求「处理」，系统将技改内容转为需求描述并调用开发模块（创建 pipeline 实例、生成设计、运行 openspec、进入编码就绪），实现技改需求到开发的全流程。
- `tech-dev-integration`: 技改与开发模块集成：技改模块复用开发模块的流水线能力（需求录入、设计生成、openspec、编码就绪）；开发模块对外提供可被调用的入口（如 API 或内部服务），供技改派发时创建实例并驱动流水线。

## Impact

- **平台前端**：新增技改模块入口与页面（任务列表、筛选、选择处理、派发结果/跳转开发实例）。
- **平台后端**：新增技改任务实体与 API；新增派发逻辑（构造需求描述、调用开发模块创建 pipeline 并触发设计/openspec 步骤）。
- **开发模块**：需暴露可被技改模块调用的接口（如「根据需求描述创建 pipeline 并返回实例 id」、「对指定实例执行设计生成与 openspec 运行」），保证现有 Web 流程不变的前提下支持程序化调用。
- **数据**：技改任务持久化（使用mysql）；技改与 pipeline 实例的关联关系需落库以便追溯。
