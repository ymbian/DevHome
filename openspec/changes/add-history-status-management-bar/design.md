## Context

开发模块目前有“提交需求 + 历史列表 + 设计页”的流程，但历史区域缺少流程级可视化状态。用户需要在多个页面与字段间来回推断“当前在哪一步”，尤其不清楚 spec 模式是否开启，导致操作重复和误判。

## Goals / Non-Goals

**Goals:**
- 给每条历史需求定义统一状态视图：`需求`、`设计`、`spec模式(开/关)`、`编码`。
- 历史区域 UI 调整为“上方状态条 + 下方历史列表”。
- 后端一次性返回可渲染状态，前端不做复杂业务推断。

**Non-Goals:**
- 不引入新的工作流引擎；沿用现有 `PipelineInstance.status` 与现有接口语义。
- 不在本次变更中实现跨用户协同/实时订阅刷新。

## Decisions

1. **状态模型决策**
   - 定义四段状态视图：
     - `requirementDone`（需求是否提交）
     - `designDone`（是否已生成设计）
     - `specModeEnabled`（该实例最近一次设计生成是否启用 spec 模式）
     - `codingReady`（是否达到可编码状态，如 `ARTIFACTS_READY/CODING_READY`）
   - spec 模式使用显式布尔字段（或可推导字段）返回，避免前端猜测。

2. **后端接口决策**
   - 在历史列表返回项中增加状态视图字段，供首页直接渲染。
   - 在详情接口也返回同样状态视图，保持页面间一致。
   - 对历史旧数据做容错映射：无新字段时按已有 `status/designContent/changePath` 推导。

3. **前端布局决策**
   - 历史区域增加 `status-bar` 容器并放在历史列表之前。
   - 点击历史任务时刷新状态条，状态条始终反映“当前选中任务”。
   - spec 模式节点显示 `ON/OFF`（中文文案可为“开启/关闭”）。

## Risks / Trade-offs

- **状态推导不一致** → 由后端集中推导，前端只展示。
- **旧数据缺字段** → 增加默认值与回退映射，避免页面空状态。
- **UI 复杂度增加** → 使用简化状态条样式，保持单行可读。
